# 검증 중심 개발 워크플로우

> **핵심 목표**: "아무 맥락 없는 AI에게 폴더 하나만 주면, 처음부터 끝까지 오류 없이 개발 완료"

---

## 절대 규칙 5가지

| 규칙 | 설명 |
|------|------|
| 1. 추측 금지 | 근거/테스트 없이 단정 ❌ |
| 2. 불확실성 표기 | 확실하지 않으면 "확실하지 않음" 명시 → 즉시 PoC 전환 |
| 3. PoC 의무 | 중/상 난이도 또는 외부 의존성 있으면 반드시 검증 |
| 4. 재현 가능성 | 누구나 동일하게 실행 가능 (환경/의존성/실행법 명시) |
| 5. 순환 반영 | PoC 결과 → 계획서 즉시 반영, 계획서 가정 → PoC로 검증 |

---

## 폴더 구조

```
/dev_poc    → PoC 코드, 테스트, 검증 기록
/dev_plan   → 최종 개발 계획서 (이것만 AI에게 전달)
```

---

## 검증 프로세스 (A→B→C→D 반복)

### A. 검증 항목 추출
계획서에서 모든 주장/가정/조건 추출 → 분류:
- ✅ 검증됨 (테스트/공식 근거 있음)
- 🧪 검증 필요 (PoC 대상)
- ❌ 불가/제약 (정책/권한/기술 한계)

### B. 우선순위 결정
프로젝트를 멈추게 하는 리스크부터 PoC:
- 인증/권한, 외부 API, 데이터 모델, 성능 병목, 배포/CI 등

### C. PoC 설계
각 PoC에 필수 포함:
- 검증 목표
- 성공 기준 (정량/정성)
- 테스트 코드
- 실행 방법
- 결과 기록 (성공/실패/원인/대안 2개 이상/다음 액션)

### D. 결과 반영
- 성공 → 계획서에 ✅ 표기
- 실패 → 원인 + 대안 + 추가 검증 계획 기록
- 모든 핵심 항목이 ✅로 수렴할 때까지 반복

---

## 완료 기준

- [ ] 모든 핵심 기능/리스크에 검증 상태(✅/🧪/❌) 명시
- [ ] 중/상 난이도는 PoC + 테스트 + 결과 존재
- [ ] 불확실 항목에 "남은 이유/검증 방법/차선책" 문서화
- [ ] 외부 입력 필요 항목 별도 문서로 완전 정리

---

## 사용자 요청 항목 관리

API 키, 권한, 환경변수 등 사용자만 할 수 있는 작업 발견 시:
`dev_plan/` 내 별도 파일(`USER_ACTION_REQUIRED.md`)로 정리

각 항목에 포함:
- **요청 이유** (왜 사용자만 가능한지)
- **요청 내용** (해야 할 작업/제공할 값)
- **단계별 절차** (구체적으로)
- **검증 방법** (성공 확인법)
- **차단 영향도** (블로커인지 아닌지)
